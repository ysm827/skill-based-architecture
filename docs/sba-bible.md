# SBA Bible：产品信念与发展方向

这份文档定义 SBA **应该成为什么**，用于判断产品方向、架构取舍和新机制是否值得进入。它不是当前功能清单，不替代代码、测试、工作流或具体计划，也不要求普通任务每次读取。

## 产品边界

> SBA 就是一个 Skill，不是 Agent 操作系统，也不是任务、状态或执行平台。

SBA 通过规则、路由、业务语义和工作流，帮助当前 Agent 更可靠地使用所在 harness 已有的 Plan、Subagent 和工具能力。某个 harness 缺少能力时，SBA 应当优雅降级；它不自建常驻服务、任务数据库、统一状态机或独立调度 runtime 来补齐平台能力。

## 最高纲领

> SBA 的目标不是让每个人学会设计 Skill，而是让普通项目成员把仓库拉下来后，Agent 就能在真实项目规则、业务语义和验证契约下可靠工作。所有能够由 SBA 作者和机制吸收的复杂度，都不应转嫁给普通使用者。

SBA 的价值不在于让 Agent 做更多动作、维护更多文件或经过更多流程，而在于提升它面对真实项目时的判断力、协调力和结果能力。

## 要证明的三件事

> 要做的不是让你变得更忙，而是要让我看到你能不能做到三件事：你能不能看到更完整的信息；你能不能在没有标准答案的时候做判断；你能不能让别人按照你的意愿做事情。

### 1. 看到更完整的信息

“更完整”不是无边界地读取一切，而是能够识别当前决定缺少什么，并沿着真实来源补齐关键上下文：

- 区分用户意图、业务语义、架构契约、代码现状、运行证据和历史结论；
- 找到 source of truth、owner、生产者到消费者的调用链和受影响路径；
- 看见矛盾、未知和证据边界，不把推测伪装成事实；
- 用渐进披露控制成本，只读取会改变下一步判断的信息。

SBA 应持续增强 Agent 建立“足够完整决策视图”的能力，而不是单纯扩大上下文窗口或堆积文档。

### 2. 在没有标准答案时做判断

真实工作经常没有唯一正确答案。Agent 不能只等待规则命中或照抄已有方案，它需要：

- 明确目标、约束、未知和完成证据；
- 区分事实判断、风险判断、产品取舍和权限边界；
- 提出真正不同的方案，说明收益、代价、可逆性和残余风险；
- 在证据足够时选择方向并推进，在缺少承重语义时准确地请求最小澄清；
- 新证据推翻前提时主动重排路径，而不是维护原计划的表面完整。

SBA 提供的是判断支架，不是用更多流程替 Agent 做决定。

新证据推翻承重结论时，不得静默覆盖；必须重新检查方案、验收、边界和 Task Anchor。涉及规范性业务判断时，由业务 owner 确认。

### 3. 让别人按照你的意愿做事情

这里的“意愿”不是操控，也不是让所有工作都经过一个中心，而是把目标变成其他 Agent、工具和协作者能够可靠执行的共同契约：

- 定义标准、角色、owner、输入、输出、禁止区和验收方式；
- 把独立工作交给合适的执行者，同时保留主线判断和最终复核；
- 让上下文来源、变更范围、验证证据和剩余风险可追踪；
- 协调冲突与资源，拒绝局部完成冒充整体结果；
- 用复核和反馈修正偏差，最终带出可验证的结果。

SBA 要让 Agent 从“自己执行得很快”进化到“能够组织可靠执行”。

## 真正的不可替代

> 真正的不可替代，不是让所有的东西都经过你，而是当信息不完整、资源不够、需求不清晰的时候，你能判断方向。

不可替代来自判断，而不是垄断流程。一个必须亲自处理所有任务、掌握所有细节才能工作的 Agent，只是瓶颈；一个能够在不确定条件下找到方向、划清边界、组织执行并对结果负责的 Agent，才具有杠杆。

> 如果你能定义标准、判断风险、协调资源、带出结果，你就不是一个执行者，你开始变成一个更大的入口。

“更大的入口”意味着更多问题可以经由它被澄清、拆解、组织和闭环，不意味着更多动作必须由它亲自完成。

## 不可妥协的原则

1. **新机制必须减少责任或解决真实失败，不能只是让架构显得完整。**
2. **默认核心应该更小但完整，而不是依赖用户理解“装哪些、不装哪些”。**
3. **规则必须能被正确激活；仅仅存进仓库不算产生价值。**
4. **Agent 的结论必须可复核：来源、边界、owner、验证契约都要清楚。**
5. **语义完整性优先于代码改动最少，但验证成本必须与风险匹配。**
6. **跨 harness 兼容应当优雅降级，不能让某个工具的限制污染整体产品模型。**
7. **SBA 负责吸收工程复杂度，用户只承担真正属于产品和业务的决策。**

## 发展方向

SBA 的演进优先服务以下方向：

1. **更完整但更克制的上下文**：增强业务语义、owner、调用关系、context provenance 和证据边界，同时减少无关读取。
2. **更强的模糊判断能力**：让 Agent 能在目标冲突、证据不足和无标准答案时给出有依据的方向，而不是机械执行最近的规则。
3. **更可靠的协作与委派**：通过结构化契约、能力边界、防递归、主 Agent 复核和结果整合，放大执行能力但不放弃责任。
4. **更低的普通用户成本**：优先删除、合并和内化复杂度；每增加一个用户可见概念，都要证明它不能由系统吸收。
5. **更安全的生命周期**：让项目知识、上游机制、生成内容和运行数据各归其主，更新可比较、可验证、可回退。
6. **从执行工具走向结果入口**：逐步具备定义标准、判断风险、协调资源和带出结果的能力，但不发展成必须承接所有工作的中心化平台。

## 明确不走的方向

- 不发展成 Agent 操作系统、任务数据库或项目管理平台；
- 不为了“以后也许有用”持续增加文件、状态、marker、脚本和术语；
- 不用结构检查的绿色结果冒充语义正确；
- 不因为外部项目设计漂亮就直接吸收，必须经过真实痛点和净收益验证；
- 不让所有事情经过主 Agent 来证明其重要性，能通过标准和契约可靠分发的工作应当分发；
- 不把维护者的安装、组合、升级和选择责任转嫁给普通使用者。

## 方向决策检查

任何重大新机制、默认行为或产品方向，在进入实现前都应回答：

1. 它让 Agent 更能看全信息、做出判断或组织别人带出结果了吗？
2. 它减少了普通使用者的责任，还是只是把作者复杂度换了一个名字交给用户？
3. 它解决的是已经出现的痛点，还是为了想象中的完整性增加机制？
4. 它的 owner、激活路径、验证契约和退出方式是否清楚？
5. 没有它时更轻的方案为什么不够？加入它后又能删除或内化什么？
6. 它是在扩大 SBA 的判断与协调杠杆，还是在制造新的中心化瓶颈？

答不清这些问题，默认不进入 SBA。

## 如何使用这份圣经

- SBA 自身进行产品定位、发展方向、外部能力吸收或重大机制设计时，先读取本文件，再进入方案规划。
- 具体 Plan 可以提出未来目标；只有实现、验证和文档同步落地后，才算成为当前产品能力。
- 本文件应保持稳定。改变最高纲领、三项能力或不可替代性的定义，必须由产品 owner 明确确认。
- 它不进入下游模板，不成为普通任务的 Always Read，也不替代更具体的规则和工作流。
